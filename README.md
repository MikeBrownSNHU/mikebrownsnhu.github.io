# Travlr Getaways — Enhancement One: Software Engineering & Design

This branch contains my first capstone enhancement for CS 499. I took the original Travlr application from CS 465 and focused on improving the code structure, security, and overall quality.

## What This Project Is

Travlr Getaways is a full-stack travel booking application built with the MEAN stack (MongoDB, Express, Angular, Node.js). It has a customer-facing site rendered with Handlebars and an Angular admin SPA where administrators can manage trip listings. Authentication uses JWT tokens.

## What I Changed and Why

During my code review in Module 2, I found a lot of issues that would be unacceptable in a real production environment. This enhancement addresses the software engineering and design problems.

### Security Fixes

The biggest issue was that the JWT authentication middleware had a bug where `next()` was called outside the verify callback. This meant every request passed through regardless of whether the token was valid. I fixed this so `next()` only runs on successful verification.

I also bumped the password hashing from 1000 PBKDF2 iterations to 210,000. The original value was set during the CS 465 course and never updated, but 1000 is way too low by modern standards. OWASP recommends at least 210,000 for SHA-512.

Other security additions:
- Added helmet middleware for HTTP security headers
- Added role-based access control (admin role required for creating/editing trips)
- Added a role field to the user model that gets included in the JWT
- Changed error messages on login to not reveal whether the email or password was wrong
- Added input validation on API endpoints so we check data before writing to the database

### Code Quality Improvements

The original code had no error handling anywhere in the API controllers. If something threw an exception, the server would just crash. I wrapped everything in try/catch blocks and return proper HTTP status codes.

I also renamed variables throughout. The original used `q` for every database result which made the code hard to follow. Now variables are named after what they actually contain (`trips`, `savedTrip`, `updatedTrip`, etc.).

Other cleanup:
- Replaced `var` with `const` throughout (the codebase mixed both)
- Removed dead scaffold code that was generated by Express but never used
- Fixed the dual error handlers in app.js that conflicted with each other
- Added JSDoc comments to document what each function does and why

### Angular Refactoring

The add-trip and edit-trip components had almost identical form templates (80+ lines of HTML each). I created a shared `TripFormComponent` that both pages use. The parent components just handle the API call while the shared component handles the form UI and validation.

I also fixed the login component which used a `setTimeout` with a 3-second delay hoping the login request would finish. This is a race condition — if the server is slow the redirect fails, and if it's fast the user waits for no reason. I replaced it with a proper Observable subscription that redirects only when we get a confirmed response.

### Schema Fixes

The trip model had `length` (number of nights) and `perPerson` (price) stored as String types. This doesn't make sense for numeric data — you can't sort or compare strings as numbers correctly. Changed both to Number with min validators. Also added a unique constraint on the trip code field since duplicates shouldn't be allowed.

## How to Run

```bash
# Install backend dependencies
npm install

# Start the Express server (port 3000)
npm start

# In a separate terminal, run the Angular admin app
cd app_admin
npm install
npx ng serve
# Admin app runs on localhost:4200
```

You'll need MongoDB running locally and a `.env` file with your `JWT_SECRET` set.
